SoulDrive 项目速读与架构讲解

这份说明基于当前源码结构整理，目标是帮助你快速建立项目全局图景：SoulDrive 是一个面向私有资产的端侧知识引擎， 当前测试与演示仍以论文作为第一切入口，但底层能力已覆盖更广泛的本地知识文档场景，重点在文档解析、增量索引、 混合检索、证据链、知识图谱、本地 GGUF 推理、Tauri 桌面打包和可移动存储授权。

后端：Python + FastAPI + Chroma + SQLite + llama.cpp 桌面：React + Vite + Tauri 2 + Rust sidecar 管理 核心数据：PDF、向量库、父子切片、FTS、图谱、审计日志 阅读日期：2026-06-15

一、先用一句话看懂

SoulDrive 的主线是：把私有知识文档放进本地或 U 盘工作区，后端离线解析并建立多种索引，用户在桌面 UI 中提问， 系统先检索可追溯证据，再让本地大模型基于证据流式回答，同时把证据链、导图和审计事件展示出来。

产品定位

面向私有资产的端侧知识引擎；论文仍然是默认演示入口。

运行边界

模型、索引、图谱和审计日志都优先落本地，API 只接受 loopback 请求。

核心差异

不只做向量检索，还加了关键词、父子切片、图谱上下文和证据门槛。

产品化方向

桌面端自动启动 sidecar，支持随机端口/token、健康监控和便携 U 盘形态。

二、总体架构图

这张图按三层阅读：上层是用户桌面体验，中层是 Python sidecar 服务和核心业务，下层是工作区数据与模型资产。 关键点是：Tauri 不直接做 RAG，只负责启动和守护后端；真正的知识处理都在 Python sidecar 内。

三、核心业务流程图

项目有两条最重要的链路：一条是知识文档入库/索引链路，另一条是用户提问后的 Hybrid RAG 链路。 理解这两条链路，基本就能看懂项目大部分代码。

四、模块拆解：每块在系统里做什么

1. 桌面壳与进程守护

souldrive-ui/src-tauri/src/lib.rs 启动 Python sidecar 或已打包的 souldrive-sidecar.exe，为每次桌面运行分配随机端口和随机 API token，并在退出时调用 /shutdown。

2. HTTP API 与安全边界

core/mcp_server.py 是接口中心；中间件负责请求体大小、loopback 限制、限流和 token 校验；公开接口只保留健康检查和运行态查询。

3. 工作区抽象

core/workspace.py 把知识文档、索引、模型、审计和运行态统一放到 SoulDrive/ 或本地 souldrive_db/ 下，避免业务代码写死盘符。

4. 文档解析与索引

core/indexer.py 管理增量索引；core/document_parser.py 用 Docling 做版面感知解析；入库时写 Chroma、父文档 SQLite 和 FTS5 关键词索引。

5. Hybrid Retrieval

core/knowledge_base.py 先做 dense 向量召回和关键词召回，再用 core/retrieval.py 的 RRF、词法重排和图谱分数合成最终证据。

6. 本地图谱

core/graph_extractor.py 使用本地 LLM 从片段抽取实体关系；core/graph_db.py 用 SQLite 存节点和边，问答时取一跳子图补充逻辑链。

7. RAG 推理引擎

core/rag_engine.py 做证据门槛、Prompt 组装、本地 GGUF 流式生成、引用覆盖审计，并在回答末尾附加 evidence/mindmap 代码块。

8. 审计与诊断

core/audit_log.py 用 previous_hash + event_hash 构造 hash-chain；core/diagnostics.py 汇总策略、模型、工作区、审计链和授权状态。

9. React 工作台

souldrive-ui/src/App.tsx 轮询运行态、指标、知识文档和审计；解析 souldrive-evidence 与 souldrive-mindmap，渲染证据卡片和可缩放导图。

五、设计思路与意图

本地优先，而不是云端 SaaS

项目把知识文档、索引、图谱、审计和模型都放在本地目录。这样牺牲的是初次解析和模型加载速度，换来的是隐私边界清楚、离线可运行和更容易做企业内网部署。

硬件和工作区是运行闸门

可移动存储不仅是数据入口，也是授权/运行态触发器。U 盘插入时解锁并索引，拔出时锁定、停 worker、释放推理资源，意图是把“数据在不在场”变成系统级状态。

检索不是单一路线

向量检索擅长语义相似，FTS/BM25 擅长关键词命中，父子切片解决“短块召回、长块回答”的上下文问题，图谱补充实体关系。这些组合起来比单纯 Chroma RAG 更稳。

回答必须能追溯

RAGEngine 在生成前做 evidence gate，回答中要求用 [E1]、[E2] 标注来源，最后把证据 JSON 返回给前端。这个设计明显是在控制幻觉和提升可审计性。

耗时任务进程隔离

索引 worker 独立进程运行，避免 PDF 解析、嵌入和图谱抽取拖死 API 主进程。Tauri 侧也有 sidecar 健康检查和自动重启。

端侧模型资源要可降级

model_runtime.py 会检测 GPU、做 isolated smoke test，失败时回退 CPU。这个设计承认用户硬件差异很大，优先保证可用性。

六、这个项目的亮点和出彩之处

亮点 为什么有价值 对应代码
端侧完整闭环 从 PDF 解析、嵌入、检索、图谱、问答到审计都在本地完成，不依赖云端 API 才能跑主链路。 core/、souldrive-ui/src-tauri/src/lib.rs
父子切片 + 混合检索 子块用于精准召回，父块用于回答上下文；dense、FTS、RRF、词法重排和图谱分共同决定证据排序。 core/parent_child_index.py、core/knowledge_base.py、core/retrieval.py
证据优先的 RAG 低证据时拒答，正常回答必须带引用编号，并把证据 JSON 给前端渲染，避免“看起来很会说但没有来源”。 core/answer_quality.py、core/rag_engine.py
知识图谱是轻量实现 没有上来引入 Neo4j/Kuzu，而是用 SQLite 存实体和关系，先把 GraphRAG 的产品价值跑通，复杂度控制得住。 core/graph_extractor.py、core/graph_db.py
便携工作区设计 本地模式和 U 盘模式共用 SoulDriveWorkspace 抽象，避免业务逻辑到处拼路径或依赖固定盘符。 core/workspace.py、core/hardware_watcher.py
桌面 sidecar 工程化 Tauri 管随机 token、随机端口、健康检查、重启和退出清理；这比“前端假设 localhost:8000 永远存在”可靠得多。 souldrive-ui/src-tauri/src/lib.rs
hash-chain 审计 审计事件有前后哈希链，可检测篡改；对本地知识系统来说，这是产品可信度的加分项。 core/audit_log.py
测试覆盖方向正确 测试不只测小函数，也覆盖路径、锁定状态、索引范围、API 安全、图谱、检索、模型运行和打包脚本。 tests/

九、当前实现边界和需要注意的地方

本地模型是硬依赖

嵌入模型 bge-small-zh-v1.5 和 GGUF 聊天模型需要在模型搜索目录里存在。缺模型时，RAG 主链路会在加载阶段失败。

图谱抽取成本较高

图谱抽取会调用本地 LLM，所以代码限制每篇文档抽取的片段数，并且只有 PRO 授权级别才会执行高级图谱抽取。

便携形态仍要关注打包体积

scripts/package-sidecar.ps1 已经在收集 Docling、OCR、Chroma、transformers、llama_cpp 等依赖后做瘦身，但 sidecar 体积和模型体积仍然是产品交付重点。

开发脚本依赖本机环境变量

运行脚本时建议显式设置 SOULDRIVE_PYTHON，并使用名为 souldrive 的 Conda 环境；不要把个人机器解释器路径写入公开文档或业务代码。

十、最短理解路线

  1. 先把它理解成“面向私有资产的本地知识引擎”，不是联网聊天应用；论文是当前默认演示入口。
  2. 再记住三件事：workspace 管数据位置，mcp_server 管接口，RAGEngine 管问答。
  3. 入库看 indexer.py，问答看 rag_engine.py，桌面启动看 src-tauri/src/lib.rs。
  4. 项目最出彩的地方是“工程闭环”：本地模型、工作区、索引、证据、图谱、审计、桌面 sidecar 都已经有明确边界。
