# SoulDrive RAG 回答质量增强设计

## 目标

降低本地知识问答中的两类问题：

1. 明明文档里有答案，但系统检索不到，误判为无证据。
2. 系统给出带引用的回答，但回答与问题弱相关，或引用证据并不能支撑结论。

本设计面向 SoulDrive 的离线端侧知识引擎场景，优先保证：

- 默认版本可在 CPU / 集显机器运行。
- 默认版本也必须具备基础幻觉抑制能力。
- 若用户自行下载更强模型并放入工作区 `models/` 目录，系统应自动识别并启用增强能力。

## 非目标

第一版不做以下能力：

- HyDE 假设性答案检索
- Step-Back 抽象回退检索
- Decomposition 多子问题拆解
- 多模型路由和复杂策略编排
- 在线 API 调用或云端模型依赖

这些能力可以在后续版本作为增强项加入，但不进入本次实现范围。

## 问题判断

当前回答质量问题并不只是“模型太小”，而是由三层共同导致：

1. 检索层：用户提问表达与文档术语不一致，导致召回不足。
2. 排序层：已有候选片段中，真正能回答问题的证据没有排到前面。
3. 生成层：模型拿着弱相关证据也会组织出看似合理但实际偏题的回答。

此外，现有 `rag_engine.py` 中回答提示词和上下文标记存在编码损坏风险，这会削弱“只能基于证据回答”的约束效果。

## 总体方案

第一版采用“轻量默认 + 自动增强”的结构：

1. 修复并重写 RAG 提示词，明确证据边界和拒答规则。
2. 在检索前增加 Multi-Query，将用户问题扩展为 2 到 3 个近义问法。
3. 在生成前增加轻量 Cross-Encoder Reranker，对候选证据进行语义重排。
4. 提高证据门控强度，低相关时拒答，中等相关时二次检索，高相关时才生成答案。
5. 默认仍使用轻量生成模型，但如果检测到更强模型存在，则自动优先使用。

## 模型策略

### 默认内置模型

- 回答模型：`qwen2.5-3b-instruct-q4_k_m.gguf`
- Reranker：`bge-reranker-base`

理由：

- 3B 模型便于 CPU 场景落地。
- `bge-reranker-base` 体积和速度相对温和，且对中英文相关性判断已明显优于纯 embedding + BM25。
- 默认版本即具备语义证据对齐能力，避免“轻量 = 容易胡说”的产品体验。

### 自动增强模型

若工作区 `models/` 目录中存在更强模型，则自动启用：

- 回答模型优先级：
  1. `qwen2.5-7b-instruct-q4_k_m.gguf`
  2. `qwen2.5-3b-instruct-q4_k_m.gguf`

- Reranker 优先级：
  1. `bge-reranker-v2-m3`
  2. `bge-reranker-base`

说明：

- 本项目主要是知识引擎，不以代码任务为核心，因此不再优先使用 coder 模型。
- 中文优先，但要兼顾英文论文与技术资料，因此生成模型与 reranker 都要具备稳定的中英双语能力。

## 检索与生成链路

### 1. Query 预处理

输入用户原始问题后，生成 2 到 3 个 Multi-Query 变体。

变体要求：

- 保持语义不变，不扩展问题范围。
- 优先改写为技术文档/论文常见表达。
- 若原问题偏口语，则生成更书面化版本。
- 若原问题是抽象问法，则生成一个更接近文档术语的版本。

### 2. 候选召回

对原问题与各个变体执行混合召回：

- dense retrieval
- keyword retrieval
- graph context retrieval

候选池在第一版中应扩大，不只取最终 `top_k`，而是先拿到更大的候选集合供 reranker 重排。

### 3. Reranker 重排

将 `(query, candidate chunk)` 输入到 cross-encoder reranker，得到每个候选片段的相关性分数。

重排后保留分数最高的若干证据用于生成。

作用：

- 压低“看起来像相关，但其实答不上问题”的片段。
- 减少带引用但弱相关的幻觉。
- 为拒答和二次检索提供更可信的阈值依据。

### 4. Evidence Gate

第一版将证据判定改为三级：

- 低证据：直接拒答。
- 中证据：触发一次二次检索。
- 高证据：允许生成。

二次检索仅进行一次，避免链路膨胀。

### 5. 回答生成

生成时只喂入 reranker 后的高质量证据。

Prompt 规则：

- 只能依据 `<context>` 回答。
- 每个关键结论必须引用 `[E1]` 这类证据编号。
- 若证据不足，只能明确说明“未找到足够可靠的相关证据”。
- 不允许利用常识补全文档未给出的事实。
- 若证据只能支持部分问题，必须显式说明回答范围有限。

### 6. 回答后校验

生成完成后做轻量回答审计：

- 检查引用是否存在。
- 检查是否至少引用了关键证据。
- 若回答无合法引用，重新生成一次。
- 若重试后仍不合格，则返回拒答。

第一版不做复杂的答案级 NLI 校验，只做可控的引用与证据覆盖审计。

## 配置与兼容性

### 自动检测

程序启动时自动检测：

- 是否存在增强版回答模型
- 是否存在增强版 reranker
- 是否具备 GPU 可用条件

若检测到 GPU，仍保持默认轻量逻辑，但在界面中提示用户可下载增强模型放入 `models/` 目录，系统会自动启用。

### 无增强模型时

系统必须保持完全可用：

- 轻量回答模型可运行
- 轻量 reranker 可运行
- 若无 GPU，则使用 CPU 推理

## 前端展示

增加一个简洁的“模型能力状态”展示区域，至少包括：

- 当前回答模型
- 当前 reranker 模型
- 当前推理设备：CPU / GPU
- 当检测到可增强条件时，提示用户可将推荐模型下载到 `SoulDrive/models/` 后自动启用

前端展示只做状态说明，不要求用户手动配置参数。

## 实现边界

本次实现仅允许在现有架构上增量改造：

- 保留当前离线 sidecar 架构
- 保留现有知识库、图谱、证据输出机制
- 不引入联网依赖
- 不引入云端模型
- 不引入大型多模型编排框架

## 测试要求

至少覆盖以下验证：

1. Multi-Query 能生成有限、稳定、语义一致的查询变体。
2. Reranker 可在存在模型时启用，在不存在模型时安全降级。
3. 弱相关证据应被 evidence gate 拒绝。
4. 强相关证据应通过 evidence gate。
5. 回答中若引用不存在或不覆盖证据，应触发重试或拒答。
6. 模型目录中存在 7B 模型时，应优先于 3B 自动使用。
7. 无 GPU、无增强模型时，主流程仍可运行。

## 分阶段实施建议

第一阶段：

- 修复 `rag_engine.py` 中提示词与上下文标签编码问题
- 抽离 prompt builder
- 强化 evidence gate

第二阶段：

- 加入 Multi-Query
- 扩大候选池
- 增加 reranker 接口与自动检测

第三阶段：

- 接入默认轻量 reranker
- 加入回答后引用审计与一次重试
- 加入前端状态展示

第四阶段：

- 调整回答模型优先级
- 增加增强模型自动启用说明
- 做完整回归测试
