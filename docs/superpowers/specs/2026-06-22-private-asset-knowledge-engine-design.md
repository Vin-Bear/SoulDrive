# 面向私有资产的知识引擎重定位设计

## 1. 目标

将当前以“论文知识工作台”为主叙事的 SoulDrive，收口为一个更贴近企业实际场景的“面向私有资产的知识引擎”。

本次改动不是重写架构，而是在保留现有端侧能力、sidecar 机制和工作区模型的前提下，完成以下目标：

- 对外产品定位从“论文工具”升级为“私有知识引擎”。
- 前端文案、README、架构讲解和提示模板统一改成通用文档/资产语义。
- API 路径和前端数据结构从 `papers/*` 收口到 `documents/*`。
- 保留内部协议前缀、环境变量、工作区根名和 Tauri 标识，避免扩大变更面。

## 2. 背景与问题

当前系统底层能力已经明显超出“论文问答”范围：

- 本地工作区管理
- 文档导入与索引
- 混合检索
- 图谱补强
- 本地模型问答
- 审计与运行态控制

但产品表达仍然强绑定学术场景：

- README 标题和描述是“端侧论文知识工作台”
- UI 使用“论文库”“篇论文”“论文归纳”等文案
- API 路径为 `/papers/list`、`/papers/import`
- 前端结构命名为 `PaperItem`、`PaperLibrary`
- 提示模板集中在论文归纳/对比

这会带来两个问题：

1. 比赛叙事偏窄，弱化了项目的企业落地空间。
2. 代码和接口命名逐渐把系统锁进“论文 vertical”，不利于后续扩展到技术文档、项目方案、运维资料、内部规范、观测记录等私有资产场景。

## 3. 非目标

本次不做以下事项：

- 不修改 `SOULDRIVE_*` 环境变量前缀。
- 不修改 `X-SoulDrive-Token` 请求头名称。
- 不修改 `SoulDriveWorkspace` 类型名。
- 不修改工作区根目录 `SoulDrive/`。
- 不修改 Tauri `identifier`、sidecar 名称、打包目录或协议级品牌字段。
- 不扩大导入格式范围，当前仍以 PDF 为主。
- 不重做索引、检索、图谱或模型推理链路。

这些部分属于运行时协议、打包标识和存储约定，收益低于改动成本，不应与本次“产品重定位”耦合。

## 4. 方案概览

采用“中度重定位”方案。

### 4.1 对外产品定位

对外统一使用以下表达：

- 主定位：`面向私有资产的知识引擎`
- 桌面副标题：`面向私有资产的端侧知识引擎`
- 保留品牌：`灵枢 SoulDrive`

这样处理的原因：

- 保留现有品牌识别，减少打包和安装侧影响。
- 将系统能力上收为“知识引擎”，避免继续被理解为狭义论文工具。
- 明确“私有资产”和“端侧”两个企业关键特征。

### 4.2 接口语义收口

将“论文接口”升级为“文档接口”：

- `GET /papers/list` -> `GET /documents/list`
- `POST /papers/import` -> `POST /documents/import`

兼容策略：

- 后端保留旧接口一段时间，内部复用同一处理逻辑。
- 前端全部切换到新接口。
- 测试新增或迁移到 `documents` 语义，旧接口测试仅保留必要兼容覆盖。

这样做可以在不破坏现有行为的前提下完成语义升级，并为后续完全移除旧路径预留空间。

### 4.3 前端数据与状态命名

前端从 `Paper*` 系列改为 `Document*` 系列：

- `PaperItem` -> `DocumentItem`
- `PaperLibrary` -> `DocumentLibrary`
- `PaperImportResponse` -> `DocumentImportResponse`
- `paperLibrary` -> `documentLibrary`
- `paperPage` -> `documentPage`
- `paperImportStatus` -> `documentImportStatus`
- `paperImportMessage` -> `documentImportMessage`

接口字段同步改名：

- `paper_count` -> `document_count`
- `papers` -> `documents`

保留字段：

- `indexed_count`
- `indexed`
- `relative_path`
- `size_bytes`
- `modified_at`

这些字段本身不带论文语义，无需改动。

### 4.4 UI 文案重写

所有用户可见文字从“论文中心”收口到“文档/知识/私有资产中心”。

重点修改项包括：

- `端侧论文知识工作台` -> `面向私有资产的端侧知识引擎`
- `论文库` -> `知识文档`
- `篇论文` -> `份文档`
- `暂无论文` -> `暂无文档`
- `更新论文索引` -> `更新知识索引`
- `导入 PDF 论文` -> `导入 PDF 文档`
- `论文归纳与智能问答` -> `私有知识问答与分析`
- `Local Paper RAG` -> `Local Knowledge Engine`
- 输入框提示从“论文归纳、对比、综述”改成“文档归纳、对比、分析、问答”
- 证据面板和空态文案从“论文问答”改成“知识问答”或“文档问答”

### 4.5 Prompt 模板重写

模板不再默认绑定学术论文，而是覆盖企业常见私有资产场景。

建议模板：

- 请归纳这批文档的核心主题、关键结论和差异，并给出引用依据。
- 请对比近三份技术资料或方案的目标、实现路径、约束和风险。
- 请生成一个围绕当前资料库的技术路线图，并标注关键来源。
- 请从企业落地角度评估这些资料对应方案的可复用性、成本和实施风险。

这样既能覆盖论文，也能覆盖技术文档、方案文档和内部知识资料。

## 5. 受影响模块

### 5.1 后端

- `core/mcp_server.py`
  - 新增 `/documents/list`
  - 新增 `/documents/import`
  - 旧 `/papers/*` 路径复用新逻辑或包装新逻辑
  - 返回字段从 `paper_*` 改为 `document_*`

- `core/paper_importer.py`
  - 文件名可以暂时保留，避免扩大变更面
  - 对外逻辑描述和调用语义转为“document import”

### 5.2 前端

- `souldrive-ui/src/App.tsx`
  - 类型名、状态名、请求路径、字段访问、UI 文案整体切换到 `documents`

- `souldrive-ui/src/App.css`
  - 选择器是否重命名取决于成本
  - 若仅类名不对外暴露，可先保留 `paper-*` 样式类，优先改 TS 和文案

### 5.3 文档

- `README.md`
- `souldrive_project_explainer.html`
- 必要时补充新的架构说明文字

### 5.4 测试

- `tests/test_mcp_server_papers.py`
  - 迁移或补充为 `documents` 语义测试
- 涉及 `paper_count`、`papers` 响应结构的测试全部同步更新

## 6. 兼容性策略

本次采用“新语义优先，旧接口兼容”的策略：

- 前端立即切到 `/documents/*`
- 后端保留 `/papers/*`，避免现有外部调用或手工测试立刻失效
- README 和测试优先展示新路径

兼容期结束后，如果项目后续确认不再需要旧接口，再单独开一轮清理。

## 7. 错误处理与边界

本次改动不改变业务行为，只改变命名和外层语义，因此错误处理规则维持不变：

- 工作区 locked 时仍返回 423
- 仅允许本地 loopback 请求
- 非授权请求仍要求 `X-SoulDrive-Token`
- 文档导入仍仅接受 PDF
- 索引流程仍然基于现有工作区和解析链路

需要额外注意的边界：

- 不要把“私有资产”文案改得过宽，导致用户误以为已支持任意二进制资产。
- 当前实际上仍以 PDF 文档为主要入口，文案应写“私有知识文档”而不是“任何资产文件”。

## 8. 测试策略

至少覆盖以下验证：

- `/documents/list` 返回结构正确
- `/documents/import` 在正常/拒绝场景下行为正确
- 前端构建通过，`App.tsx` 类型改名无残留
- 旧 `/papers/*` 接口兼容测试仍通过或保留最小验证
- README 与讲解 HTML 中的产品定位已切换为“私有资产知识引擎”

建议执行：

```powershell
python -m unittest discover -s tests -v
cd souldrive-ui
npm run build
```

## 9. 实施顺序

1. 先改后端接口与返回字段，建立 `documents` 语义，并保留 `papers` 兼容层。
2. 再改前端类型、状态、请求路径和显示文案。
3. 再改 README、架构讲解 HTML、提示模板。
4. 最后收口测试与回归验证。

## 10. 预期结果

改完之后，项目会从：

- `端侧论文知识工作台`

升级为：

- `面向私有资产的端侧知识引擎`

同时保留现有端侧 RAG、图谱补强、审计、sidecar、工作区和本地模型能力不变。

这次改动的本质是：

- 提升产品定位
- 统一系统语义
- 扩大场景解释力

而不是重做底层架构。
